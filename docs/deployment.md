# Deployment Guide

This app is a full-stack retail agent experience:

- React/Vite frontend
- Express API backend
- OpenAI Responses API integration
- Local mock product, policy, and knowledge data
- Optional PostgreSQL telemetry
- Optional GA4 browser events, Measurement Protocol forwarding, and Data API reporting

It can be deployed either as one Vercel project or as one long-running Node service. Vercel is the best fit for the hackathon demo because the browser, API, mock data, analytics routes, product images, and GLB models stay on the same domain.

## Vercel Deployment

Vercel serves the Vite client from `dist/client` and runs the Express API through `api/[...path].ts`.

```text
vercel.json              # Vercel build, function, and rewrite config
api/[...path].ts         # Serverless adapter for Express API routes
src/server/app.ts        # Shared Express app factory
src/server/index.ts      # Local/traditional Node server listener
src/client/              # React/Vite app
public/                  # Static images, GLB product models, favicon assets
data/                    # Mock catalog and knowledge files included in the API function
dist/client/             # Generated by npm run vercel-build
dist/server/             # Generated only for traditional Node hosting
```

Use these settings in Vercel:

```text
Framework Preset: Vite
Install command: npm ci
Build command: npm run vercel-build
Output directory: dist/client
Node.js runtime: Node.js 22.x
```

The checked-in `vercel.json` already sets the build command, output directory, API function include files, and rewrites:

```text
/health      -> /api/health
/api/(.*)    -> /api/$1
/(.*)        -> /index.html
```

The final rewrite enables React single-page app routes without breaking `/api/*`.

## Vercel Environment Variables

Set these in Vercel Project Settings > Environment Variables:

```bash
OPENAI_API_KEY=sk-...
OPENAI_MODEL=gpt-5.5
OPENAI_VECTOR_STORE_ID=vs_...
BRAND_NAME=Aster & Ridge
CLIENT_ORIGIN=https://your-vercel-domain.vercel.app
```

Leave `VITE_API_BASE` empty or unset for the normal same-domain deployment.

Optional durable telemetry:

```bash
DATABASE_URL=postgres://...
```

Without `DATABASE_URL`, the demo still works with in-memory telemetry, but events reset on function cold starts or redeploys.

Optional GA4 browser and server-side diagnostics:

```bash
GA4_MEASUREMENT_ID=G-XXXXXXXXXX
GA4_API_SECRET=your_measurement_protocol_secret
VITE_GA4_MEASUREMENT_ID=G-XXXXXXXXXX
VITE_GA4_DEBUG_MODE=true
GA4_DEBUG_MODE=true
```

Optional GA4 dashboard readback:

```bash
GA4_PROPERTY_ID=123456789
GA4_CLIENT_EMAIL=service-account@project.iam.gserviceaccount.com
GA4_PRIVATE_KEY="-----BEGIN PRIVATE KEY-----\n...\n-----END PRIVATE KEY-----\n"
```

Or use:

```bash
GA4_SERVICE_ACCOUNT_JSON={"type":"service_account",...}
```

Do not set `PORT` or `HOST` for Vercel. Those are only for local and traditional Node service deployments.

## Vercel Deploy Commands

From the repo root:

```bash
npm ci
npm run vercel-build
npx vercel login
npx vercel
npx vercel --prod
```

After the first `npx vercel` run, review the generated `.vercel/project.json` locally. Do not commit `.vercel/`.

## Vercel Smoke Test

Open:

```text
https://your-vercel-domain.vercel.app/health
```

Expected response:

```json
{
  "status": "ok",
  "mode": "live_openai",
  "vectorStoreConfigured": true
}
```

Then verify:

1. Open the app URL.
2. Ask the chat for product recommendations.
3. Confirm product cards load images from `/images/products/...`.
4. Open the 3D product viewer and confirm GLB models load from `/models/products/...`.
5. Open Analytics and send a demo GA4 event if GA4 is configured.

## Other Hosting Options

Good long-running Node choices for this POC:

- Render Web Service
- Railway Service
- Azure App Service
- Fly.io
- AWS App Runner
- Any VM or container service that can run Node 22

Use these settings for a traditional Node service:

```text
Runtime: Node.js 22.x
Build command: npm ci && npm run build
Start command: npm start
```

The full build produces:

```text
dist/client    # React frontend
dist/server    # Express API bundle
```

The production server serves both from the same process.

## Traditional Node Environment Variables

Set these in the hosting platform's environment/secret settings:

```bash
NODE_ENV=production
OPENAI_API_KEY=sk-...
OPENAI_MODEL=gpt-5.5
OPENAI_VECTOR_STORE_ID=vs_...
BRAND_NAME=Aster & Ridge
```

Most platforms inject `PORT` automatically. If your platform does not, set:

```bash
PORT=8787
```

Use this only when needed by your host:

```bash
HOST=0.0.0.0
```

Set `CLIENT_ORIGIN` to your production URL when the API is called from another frontend domain. If the Express server serves the frontend directly, same-origin requests work without a separate `VITE_API_BASE`.

```bash
CLIENT_ORIGIN=https://your-production-domain.com
```

## PostgreSQL

For durable analytics/profile storage, provision PostgreSQL and set:

```bash
DATABASE_URL=postgres://...
```

Apply the schema once from a secure environment:

```bash
npm run db:schema
```

Without `DATABASE_URL`, the app still runs, but telemetry is in memory and resets when the service restarts.

## Separate Frontend/API Deploy

If you host the frontend separately from the API:

Frontend build-time variable:

```bash
VITE_API_BASE=https://your-api-domain.com
```

API runtime variable:

```bash
CLIENT_ORIGIN=https://your-frontend-domain.com
```

For the normal Vercel same-domain deployment, leave `VITE_API_BASE` empty.
